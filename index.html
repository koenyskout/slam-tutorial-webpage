Start Here

What SLAM Is, Without Robotics Jargon

SLAM means: build a map while figuring out where you are inside that same map. That is hard because the robot's motion estimate is noisy, and sensors are noisy too.

A useful everyday analogy is walking in a dark building with a weak flashlight and a noisy step counter. You are constantly combining: "How far I think I moved" + "What I think I just saw."

In robotics language, "where I am" is localization, and "what the world looks like" is mapping. SLAM couples them: better localization gives better maps, and better maps improve localization.

How To Use This Chapter

  • Change one control at a time so cause and effect stay clear.
  • Watch both the map and the on-canvas overlays; overlays name what changed.
  • When a result looks wrong, pause and drag controls back toward low noise.

What Colors Usually Mean

  • Blue: reference or hidden truth.
  • Orange/gray: noisy or uncertain intermediate estimates.
  • Green: corrected estimate after fusing information.
Two common map types. This chapter covers both: sparse landmarks (discrete points) and occupancy grids (cell-wise free/occupied probabilities).

Overview

The SLAM Recursion: Predict, Correct, Repeat

SLAM runs the same loop many times per second: predict where the robot went, then correct that prediction with sensor data. Over time, this loop estimates both robot pose and map.

\[ p(x_k,m \mid z_{1:k},u_{1:k}) \propto p(z_k \mid x_k,m) \int p(x_k \mid x_{k-1},u_k)\,p(x_{k-1},m \mid z_{1:k-1},u_{1:k-1})\,dx_{k-1} \]

If this equation looks heavy, treat it as bookkeeping: \(u_k\) is motion input, \(z_k\) is sensor data, \(x_k\) is pose, and \(m\) is map. The next sections unpack these symbols one by one, then rebuild them into full SLAM systems.

Notation Cheat Sheet

  • \(x_k\): robot pose at time step \(k\), usually \((x,y,\theta)\) in 2D.
  • \(u_k\): control or odometry input between \(k-1\) and \(k\).
  • \(z_k\): sensor measurements at time \(k\) (ranges, bearings, scan points, etc.).
  • \(m\): map state (landmarks or occupancy cells).
\[ \begin{aligned} \text{Predict: } & \hat{x}_k^- = f(\hat{x}_{k-1}, u_k), \quad P_k^- = F_k P_{k-1} F_k^\top + Q_k\\ \text{Correct: } & K_k = P_k^- H_k^\top (H_k P_k^- H_k^\top + R_k)^{-1},\\ & \hat{x}_k = \hat{x}_k^- + K_k (z_k - h(\hat{x}_k^-)), \quad P_k = (I - K_k H_k)P_k^- \end{aligned} \]

This is the EKF form of the same recursion. You do not need to memorize it immediately. Use it as a map: motion model \(f\), measurement model \(h\), and uncertainty matrices \(Q_k\), \(R_k\), \(P_k\).

Bridge Example

One Complete SLAM Update Cycle, Equation by Equation

Before we dive into each foundation topic in detail, run one concrete SLAM cycle end-to-end. This example shows how the equations become an implementation and how that implementation moves the estimate toward truth.

\[ \begin{aligned} \text{(1) Predict state: }& \hat{x}_k^- = f(\hat{x}_{k-1}, u_k)\\ \text{(2) Predict covariance: }& P_k^- = F_k P_{k-1} F_k^\top + Q_k\\ \text{(3) Gain: }& K_k = P_k^- H_k^\top (H_k P_k^- H_k^\top + R_k)^{-1}\\ \text{(4) Correct: }& \hat{x}_k = \hat{x}_k^- + K_k\big(z_k - h(\hat{x}_k^-)\big) \end{aligned} \]

Reference Implementation (Used By The Visualization)

const xPred = f(x, u);                     // Eq. (1)
const PPred = F(x, u) @ P @ F(x, u).T + Q; // Eq. (2)
const K = PPred @ H(xPred).T @ inv(H(xPred) @ PPred @ H(xPred).T + R); // Eq. (3)
const xPost = xPred + K @ (z - h(xPred));  // Eq. (4)
P = (I - K @ H(xPred)) @ PPred;

Part I

Foundations: Representation, Motion, and Uncertainty

Sections 1-4 build the mathematical core used by all SLAM variants: coordinate transforms, motion prediction, covariance, and Bayesian fusion.

Section 1

State, Frames, and Coordinate Transforms

Before mapping, the robot must express everything in one coordinate system. In 2D we write pose as \(\mathbf{x} = (x, y, \theta)\): location \(x,y\), and facing direction \(\theta\).

Sensors usually report points relative to the robot ("2 meters ahead"), not in map coordinates. This equation converts local sensor coordinates into world/map coordinates:

\[ \begin{aligned} x_w &= x + p_x\cos\theta - p_y\sin\theta \\ y_w &= y + p_x\sin\theta + p_y\cos\theta \end{aligned} \]

This transform is the base layer under almost every SLAM measurement update.

Pipeline Role

  • Input: robot pose plus sensor point in robot-local coordinates.
  • Output: map-frame coordinates that can be compared with landmarks or grid cells.

Try This In The Figure

  • Drag the heading handle while keeping robot position fixed.
  • Then drag the transformed point and watch local coordinates change.
  • Notice how the same physical point can have different coordinates in different frames.
Why this matters. Most SLAM errors are ultimately transform or frame-consistency errors.
Next step. Once coordinates are consistent, we can model robot motion and predict where it went.

Section 2

Prediction from Odometry

Odometry is "where I think I moved" from wheel encoders or motion sensors. If the robot moves by \(\Delta s\) and turns by \(\Delta \theta\), one common update is:

\[ \begin{aligned} x_k &= x_{k-1} + \Delta s \cos(\theta_{k-1}) \\ y_k &= y_{k-1} + \Delta s \sin(\theta_{k-1}) \\ \theta_k &= \theta_{k-1} + \Delta\theta \end{aligned} \]

Even tiny per-step errors accumulate. This is called drift. The visualization shows how pure prediction gradually walks away from truth.

Pipeline Role

  • Input: previous pose estimate and odometry/control increment.
  • Output: predicted pose \( \hat{x}_k^- \), before any sensor correction.

Try This In The Figure

  • Set noise low and watch true and odometry paths stay close.
  • Increase noise slowly and observe the gap line between true and odom.
  • Increase speed and notice drift grows faster over distance.
Core limitation. Prediction-only localization is unbounded in error over long trajectories.
Next step. To correct drift rigorously, we need a mathematical representation of uncertainty.

Read the drift number while moving the noise and speed sliders. This is your first "prediction vs reality" gap.

Section 3

Uncertainty as Covariance Ellipses (EKF Core Geometry)

In many SLAM systems, uncertainty is represented by a Gaussian belief: mean pose \( \boldsymbol{\mu} \) and covariance matrix \( \Sigma \). In 2D position, \( \Sigma \) becomes an ellipse: orientation shows correlated error, axis lengths show uncertainty magnitude.

\[ p(\mathbf{x}) = \mathcal{N}(\mathbf{x};\boldsymbol{\mu},\Sigma), \qquad (\mathbf{x}-\boldsymbol{\mu})^\top \Sigma^{-1} (\mathbf{x}-\boldsymbol{\mu}) = \chi^2_{2,\alpha} \]

EKF fusion in information form is:

\[ \Sigma^+ = \left((\Sigma^-)^{-1} + R^{-1}\right)^{-1}, \qquad \boldsymbol{\mu}^+ = \Sigma^+ \left((\Sigma^-)^{-1}\boldsymbol{\mu}^- + R^{-1}\mathbf{z}\right) \]

Here, blue is ground-truth state, gray is prior belief \((\boldsymbol{\mu}^-, \Sigma^-)\), orange is measurement model \((\mathbf{z}, R)\), and green is fused posterior belief.

Pipeline Role

  • Input: prior mean/covariance and measurement mean/covariance.
  • Output: posterior mean/covariance with explicit confidence geometry.

Try This In The Figure

  • Drag blue/gray/orange means to create disagreement between truth, prediction, and measurement.
  • Stretch a major axis to increase uncertainty in one direction.
  • Rotate ellipses and watch how posterior orientation changes with anisotropic uncertainty.
Next step. Now we apply this fusion logic to a concrete nonlinear sensor model.

Blue handle = truth state. Gray/orange handles set prior and measurement means/covariances: center point = mean, long-axis = principal variance+angle, short-axis = orthogonal variance.

Section 4

Probabilistic Fusion: Prior + Measurement = Posterior

In SLAM, we do not keep one exact position. We keep a set of plausible positions with confidence values. Blue is ground-truth pose (teaching reference), gray is prior belief, orange is measurement likelihood, and green is fused posterior belief.

The map below now uses a nonlinear range-bearing measurement model around one known landmark. Orange shows where the robot could be according to that measurement (an annular sector), blue is ground truth, gray is prior belief, and green is the fused posterior.

\[ p(\mathbf{x}\mid z,m) \propto p(z\mid \mathbf{x},m)\,p(\mathbf{x}) \]

with likelihood

\[ p(z\mid \mathbf{x},m)\propto \exp\!\left( -\frac{1}{2}\left[ \left(\frac{\|\mathbf{x}-\mathbf{\ell}\|-r}{\sigma_r}\right)^2 + \left(\frac{\mathrm{wrap}(\mathrm{atan2}(y-\ell_y,\;x-\ell_x)-\phi)}{\sigma_\phi}\right)^2 \right] \right) \]

Lower \(\sigma_r,\sigma_\phi\) makes the orange likelihood narrower, so posterior contracts and shifts more aggressively. Higher uncertainty broadens it, so prior dominates.

Pipeline Role

  • Input: prior belief and one measurement model around a landmark.
  • Output: updated posterior belief over robot pose.

Try This In The Figure

  • Drag the blue truth pose and landmark to move measurement geometry.
  • Increase range/angle uncertainty handles and watch green posterior spread out.
  • Tighten uncertainty and watch posterior collapse toward a smaller region.
Next step. We now put this update into a running trajectory with repeated motion and observations.

Blue controls truth pose. Gray controls prior belief. Orange controls measurement model. Green is fused belief.

Part II

Landmark SLAM Pipeline: Correct, Associate, Jointly Estimate

Sections 5-8 turn the foundations into a working landmark-SLAM system, then reinterpret that system as a global optimization problem.

Section 5

Measurement Update with Known Landmarks

Here we assume landmark positions are known. The robot predicts motion, then corrects pose using landmark observations.

\[ \begin{aligned} \hat{x}_k^- &= f(\hat{x}_{k-1}, u_k) \\ \hat{x}_k &= \hat{x}_k^- + K_k\left(z_k - h(\hat{x}_k^-)\right) \end{aligned} \]

In EKF, \(K_k\) is computed from Jacobians: \(F_k=\frac{\partial f}{\partial x}\), \(H_k=\frac{\partial h}{\partial x}\), and noise covariances \(Q_k, R_k\). The key idea remains geometric: use measurements to pull the predicted pose toward sensor-consistent locations.

Geometrically, landmark measurements pull the drifting estimate back toward plausible positions.

Pipeline Role

  • Input: predicted pose plus measurements to known landmarks.
  • Output: corrected pose after applying innovation-weighted update.

Try This In The Figure

  • Raise odometry noise and keep sensor noise low.
  • Then increase correction gain and watch corrected path re-align.
  • Set sensor noise high and notice over-correction risk if gain is too large.
Tuning insight. Gain trades responsiveness for noise sensitivity.
Next step. This only works if each measurement is matched to the correct landmark.

Practical tuning rule: increase gain until correction helps, then back off when noise causes jitter.

Section 6

Data Association: Matching Measurements to Landmarks

A sensor reading is only useful if you attach it to the correct landmark. This matching step is called data association.

This figure samples noisy readings from one true landmark and asks: which candidate landmark best explains this reading?

\[ p(z\mid \ell_i) \propto \exp\left(-\frac{1}{2}\Big[\big(\tfrac{\Delta r_i}{\sigma_r}\big)^2 + \big(\tfrac{\Delta \phi_i}{\sigma_\phi}\big)^2\Big]\right) \]

Practical systems add gating (reject unlikely matches), and often compare several hypotheses (nearest-neighbor, JCBB, or probabilistic assignment). This section shows the core likelihood mechanism behind those methods.

Pipeline Role

  • Input: one noisy measurement and a set of candidate landmarks.
  • Output: association decision (which landmark ID to update).
Failure mode. Wrong associations can corrupt both map and trajectory even when other components are correct.

Try This In The Figure

  • Drag landmarks close together so geometry becomes ambiguous.
  • Increase noise and watch wrong-association rate rise.
  • Move landmarks farther apart and see probabilities separate again.
Next step. With prediction, correction, and association in place, we can run full online SLAM.

Section 7

Online SLAM: Estimating Pose and Map Together

This is the full coupled problem: estimate robot pose and map landmarks at the same time. Better pose gives better map updates, and a better map gives better pose corrections.

\[ p(x_{0:k}, m \mid z_{1:k}, u_{1:k}) \]
\[ \mathbf{y}_k= \begin{bmatrix} \mathbf{x}_k\\ \mathbf{m}_1\\ \vdots\\ \mathbf{m}_N \end{bmatrix}, \qquad P_k= \begin{bmatrix} P_{xx} & P_{xm}\\ P_{mx} & P_{mm} \end{bmatrix} \]
\[ p(x_{0:k},m \mid z_{1:k},u_{1:k}) \propto p(m)\,p(x_0)\,\prod_{t=1}^k p(x_t\mid x_{t-1},u_t)\,\prod_{t=1}^k p(z_t\mid x_t,m) \]

This joint posterior (trajectory \(x_{0:k}\) and map \(m\)) is the central SLAM object. The visualization uses a simplified online variant with landmark refinement and loop-closure-like corrections.

The off-diagonal covariance blocks (\(P_{xm}, P_{mx}\)) encode robot-map correlations. Those correlations are why observing one landmark can improve many related estimates.

Pipeline Role

  • Input: motion updates + associated landmark observations over time.
  • Output: jointly consistent pose estimate and landmark map estimate.
Modeling simplification. This teaching demo assumes known landmark identities after association and uses lightweight correction rules instead of full sparse graph optimization or full EKF covariance propagation.
About error numbers. Pose/map errors shown in the canvas overlays are evaluation metrics against hidden simulation truth. Real robots do not have this ground-truth stream online.

Try This In The Figure

  • Increase odometry noise and watch pose error rise.
  • Increase correction gain to recover, then raise sensor noise to see the trade-off.

Section 8

Pose Graphs and Loop Closure Optimization

Another major SLAM formulation is graph-based SLAM. Each robot pose is a node. Each odometry or sensor relation is an edge (constraint). Loop closure adds a long-range edge that says: "this late pose should be consistent with an earlier place."

\[ \mathbf{x}^\star = \arg\min_{\mathbf{x}} \sum_{(i,j)\in\mathcal{E}} \left\| \mathbf{e}_{ij}(\mathbf{x}_i,\mathbf{x}_j) - \hat{\mathbf{z}}_{ij} \right\|_{\Omega_{ij}}^2 \]

The objective above says: find all poses \(\mathbf{x}\) that best satisfy all constraints (weighted by information matrix \(\Omega_{ij}\)). In practice this is solved iteratively (Gauss-Newton, Levenberg-Marquardt, iSAM variants).

Pipeline Role

  • Input: a trajectory graph with odometry edges and loop-closure edges.
  • Output: globally optimized trajectory where accumulated drift is redistributed.

Try This In The Figure

  • Start optimization and watch green trajectory globally deform to satisfy closure.
  • Increase loop-closure weight to enforce stronger global consistency.
  • Use single-step mode to see optimization as many tiny corrections, not one jump.
Next step. So far the map was sparse landmarks. Next we switch to dense occupancy mapping.

Part III

Dense Mapping and Physical Realism

Sections 9-10 switch from sparse landmarks to occupancy grids, then add kinematic constraints so the estimator remains physically plausible.

Section 9

Occupancy Grid Mapping from Range Beams

Not all SLAM maps are sparse landmarks. A common alternative is occupancy grids, where each cell stores probability of being occupied. Range beams update free space along the ray and occupied space at the hit.

\[ L_t(c) = L_{t-1}(c) + \begin{cases} +\lambda_{occ}, & \text{if beam hits cell } c \\ -\lambda_{free}, & \text{if beam passes through } c \end{cases} \]
\[ p_t(c)=\sigma(L_t(c))=\frac{1}{1+e^{-L_t(c)}}, \qquad \mathcal{F}_t=\left\{c\;\middle|\; L_t(c)\lt L_{\mathrm{free}},\; \exists n\in\mathcal{N}(c):\left|L_t(n)\right|\lt\tau_{\mathrm{unk}}\right\} \]

Left panel: estimated occupancy from noisy beams. Right panel: ground-truth environment (hidden from the estimator, shown only for teaching). Over time, repeated scans reduce uncertainty and reveal structure. The planner drives toward nearest reachable frontier cells (orange markers), then stops when no reachable frontier remains.

Pipeline Role

  • Input: repeated range beams along robot trajectory.
  • Output: probabilistic free/occupied grid map and frontier-driven exploration targets.

Try This In The Figure

  • Pause and inspect orange frontier dots: those are boundaries between known free space and unknown space.
  • Increase beam count to make map discovery faster.
  • Increase sensor noise and watch uncertainty remain gray for longer.
Next step. Final layer: enforce motion realism so corrections remain physically plausible.

White means confident free space, dark means confident occupied, light gray means still uncertain.

Section 10

Kinematic Constraints: Preventing Non-Physical Pose Jumps

Real robots cannot teleport or spin infinitely fast. So even during correction, state updates should obey motion limits.

\[ \dot{x}=v\cos\theta,\quad \dot{y}=v\sin\theta,\quad \dot{\theta}=\omega,\quad |\omega|\le \omega_{\max},\; |\dot{v}|\le a_{\max} \]

The green estimate applies bounded yaw-rate and acceleration. The dashed gray baseline applies unconstrained corrections. Fixed blue landmark dots are ground truth, while green landmark dots/circles are the moving map estimates. Compare jump metrics to see why motion priors improve physical plausibility.

Pipeline Role

  • Input: unconstrained correction proposal from estimator.
  • Output: kinematically feasible state update for real robot execution.

Try This In The Figure

  • Enable the raw baseline and compare raw vs kin jump metrics.
  • Reduce max yaw-rate and see motion become smoother but less responsive.
  • Increase correction gain and check whether kin stays close without large jump spikes.

Compare "raw" and "kin" jump values. A good setting keeps kin smooth without letting error grow too much.

Closing Notes

What You Learned, and What Production SLAM Adds

You now saw the full SLAM stack from first principles: state representation, motion prediction, Bayesian fusion, data association, map updates, loop closure, and physical motion constraints.

Key Ideas To Keep

  • SLAM is a probabilistic inference problem, not just geometry.
  • Most failures come from a mismatch between uncertainty and model assumptions.
  • Loop closure and consistent data association are critical for long-term accuracy.
  • Good motion models prevent impossible corrections that break physical plausibility.
Beyond this chapter. Production systems add robust outlier rejection, richer sensor models (LiDAR scan matching, visual features, IMU preintegration), sparse linear solvers, and careful real-time systems engineering.

These visualizations are pedagogical simplifications of EKF-SLAM, graph-SLAM, and occupancy-grid mapping. They are designed for conceptual understanding and mathematical intuition rather than production-grade estimation.
